# /spec — 決めることを前倒しで固め、承認後は迷わず実装する

仕様フェーズの目的は AI の自立性を上げること。聞くべきことは実装前に全部聞き、スペックを
**「新規セッションの AI にこのファイルだけ渡しても、追加の質問なしで完成させられる指示書」**に仕上げる。
承認後の実装中に(不可逆な操作・本人にしか決められない分岐を除いて)質問が出たら、それは仕様フェーズの失敗として記録する。

発動の目安: 複数ファイル/半日以上の作業・本人にしか決められない分岐が2つ以上・やり直しコストが高い、のいずれか。
ワンライナー級の小修正・明らかな単発作業には使わない（過剰プロセス）。そのまま実装する。
要件が会話で既に具体化済みなら**軽量パス**: R1〜R3 を省略して起草（ゲートA含む）に直行し、インタビューの代わりにプレイバック+ゲートBのみ行う。R3 必須質問だけは省略不可 — ゲートA通過後、全文提示の直前に単独1問を AskUserQuestion で出し、回答を §12 に記入してからゲートBを出す。

## 手順

### 0. 引数とモード判定
- `/spec <一行概要>` — 概要をシードに手順1から開始。
- `/spec <既存スペックのパス>`（または既存スペックと同じ機能名）— 改訂モード（→「改訂（再実行）」節）。機能名一致の照合は手順1で保存先を確定した後に、その `specs/` と `~/specs/` を対象に行う（照合前は通常モードで開始してよい）。
- 引数なし — 直近の会話文脈から対象を推定。推定できなければ最初の AskUserQuestion で「何を作るか」から聞く（候補＋自由入力）。
- **出力言語の決定（最初に一度）:** スキルディレクトリの `config.yml` を読む（無ければ `output_language: auto` とみなす）。この値が、インタビューの質問・プレイバック・仕様書(§1〜13)の自然文の言語を決める — `ja`=日本語 / `en`=英語 / `auto`=会話・対象プロジェクトの主言語に合わせる。コード・コマンド・パス・slug・技術用語は値に関わらず常に英語。AskUserQuestion の選択肢ラベルもこの言語で書く。言語を変えたいユーザーには `config.yml` の1行を編集して新セッションで開くよう案内する。

### 1. 自己調査 — 質問より先（質問は最後の手段）
**調べれば分かること・会話で既に答えられていることを聞くのは停滞。** 質問の前に必ず:
1. **会話文脈を浚う。** 既に答えられている要件・制約・好みを抽出し、再質問禁止リストに載せる。抽出した既出回答は起草時に §12 の D-n として日付付きで記録する（会話既出 = 本人決定。セッションを跨ぐ再質問防止はこの記録が担保する）。
2. **対象プロジェクトを特定し読む。** git root・既存 `specs/`・CLAUDE.md・README・主要コード。
   技術スタック・既存の流儀・再利用できる資産はここで確定する（聞かない）。既存 `specs/` に同一機能を扱うスペックがあればそれを正として改訂モードに切り替える（slug の表記揺れで新規ファイルを作らない）。
   新規構想で読む対象が無い場合は、環境制約とプラットフォーム標準（例: macOS 常駐 → Swift/SwiftUI）から技術スタックを T 仮置き（根拠1行）し、ゲートBの全文提示で拒否権を渡す。
3. **環境制約を確認する。** グローバル運用ルール・メモリに既出の制約（予算・禁止技術・マシン構成等）は質問せず仕様の前提に置く。
4. **外部の事実は自分で調べる。** 技術的な実現可能性・制約（ライブラリが目的の機能を持つか、API の上限・料金、SDK/プラットフォームの仕様、バージョン差異など）は WebSearch・WebFetch（公式ドキュメント検索の MCP があればそれも）で確認してから仕様に反映する。「〜はできますか？」と本人に聞くのは停滞 — **事実は調べ、決定（意向・好み・優先順位）だけを聞く**。調べた外部事実は §9/§10 に出典を1行添えて記録する。
5. **残った疑問を濾過する。** 「本人にしか決められないもの」だけ残し、各々に推奨デフォルト案を用意する。

スペックファイルの保存先もここで確定: 対象プロジェクト root 直下 `specs/<kebab-slug>.md`（`specs/` が無ければ作る）。
プロジェクト未定・新規構想なら `~/specs/`（無ければ作る）にフォールバック。後でプロジェクト dir ができたら mv して §13 に記録し、報告は完了報告にまとめる（新規 dir 名は slug と揃えた T 仮置き）。
slug は英小文字ケバブ。日本語の機能名は意味を英訳して slug 化する（例: 在庫アラート → `inventory-alert`）。

### 2. インタビュー — 全質問を AskUserQuestion で出す
**裸のテキスト質問は出さない。すべての質問は AskUserQuestion ツールで出す** — 構造化ダイアログとして表示され、通知連携（リモート/モバイルで応答する構成）にも乗るので離席中でも選んで答えられる（裸のテキスト質問は通知に乗らず、離席中のユーザーに届かない）。
- 自由記述で答えたい核心質問（目的と成功の定義・フローの実況・入出力の実例・「絶対に起きてはいけない事故」）も、推奨案＋代表的な答えの型を選択肢にした AskUserQuestion で出す。ユーザーは選択肢を選んでも、自動付与される自由入力欄に自分の言葉で書いてもよい — 質問文に「近いものを選ぶか、自由入力で詳しく」と一言添える。
- 選択肢が自然に列挙できる分岐（操作形態・異常時方針・保存/配布形態・優先順位など）はそのまま選択肢にする。
- **1ラウンド = AskUserQuestion 1回（最大4問）。** 依存しない質問はまとめて1回で出し、ラウンド数を減らす（依存する質問は次ラウンドへ）。

AskUserQuestion の規律:
- 1回最大4問・各2〜4択・header は12文字以内。**「その他」「なし」を選択肢に入れない**（自由入力欄は自動付与され、ユーザーは回答せずスキップもできる）。
- 推奨案は先頭に置き label 末尾に「(推奨)」。複数選択可の問いは multiSelect: true + 質問文も複数選択前提の言い回しにする。
- 見た目・UI の分岐は preview にモック（ASCIIレイアウト等）を添えて見比べさせてよい（preview は単一選択の質問のみ。環境が未対応ならモックを通常テキストで先に提示する）。
- 答えが他の質問の前提になる依存質問は同じ呼び出しに入れない（次ラウンドに回す）。
- 自由記述回答・会話文脈で既に答えが出た項目は出さずにスキップする。
- 各質問を「答えによって設計が変わるか？」で篩にかける。変わらないなら聞かずに仮置き(T)にする。
- AskUserQuestion が使えない環境では、番号付きテキストで選択肢+推奨デフォルトを提示して代替する。

ラウンド構成（粗→細・不可逆な事項ほど先。回答で疑問が解消すれば短縮する）:
| R | 確定するもの | 中身（→記録先テンプレ§） |
|---|---|---|
| 1 | 方向: 目的・文脈・境界・即死条件 | 成功の定義 / 誰がどんな場面で使うか / v1スコープと「やらないこと」 / 禁止事項・期限・環境（→§1·2·3·9） |
| 2 | 骨格: 機能の核 | 入出力形式（実例をもらう） / 主要フローの実況 / 永続データと消えたら困るもの / 既存資産との統合範囲（→§4·5·6·10） |
| 3 | 輪郭: 異常系・非機能・検証 | 想定外入力時の方針と最悪事故 / 非機能の数値閾値（秒・件数・円） / 受け入れシナリオ / 仮置きの裁量範囲（→§7·8·11·12） |
| 4 | 閉じ: 齟齬ゼロ | 新規質問なし。「私の理解では〜」で全回答+仮置き一覧をプレイバック。疑問が残っていなければ R4 はゲートBの全文提示に統合してよい |

回答の取り回し:
- 各ラウンドの末尾に理解の要約を1段落返し、齟齬をその場で潰す。
- 回答が他の観点を先回りして答えていたら、その観点の質問を落とし再質問禁止リストに即追加する。
- 自由入力（その他）での回答が複数論点を跨ぐときは観点ごとに分解してプレイバックに載せ、どの観点への回答として記録したかを明示する。
- 回答が曖昧なら、選択肢・具体例に言い換えて**1回だけ**再質問する。なお曖昧なら推奨案を T 仮置きと明示して前進する。
- 質問で返されたら（「どっちがいいと思う？」）、推奨案+理由を即答し「異論がなければこれで確定します」と添えて前進。次のプレイバックに載せ、訂正がなければ D として記録する（同じ質問を出し直さない）。
- R3 では必ず AskUserQuestion で聞く: 「細部は私が仮置きで進めます。**ここだけは勝手に決めるな**という領域はありますか？」（選択肢例: 「特になし・任せる (推奨)」「ある → 自由入力で指定」）— 回答はテンプレ§12 の「勝手に決めるな領域」に記録し、手順5 の質問基準になる。
  **ラウンド短縮時・「任せる」宣言後も、この1問だけは単独で必ず聞く**（委任範囲そのものの確認であり省略不可）。Skip・無回答なら §12 に「なし（本人 Skip YYYY-MM-DD）」と出所付きで記録する。
- ユーザーが「任せる」と言ったら、残りの観点（後続ラウンド含む）は推奨デフォルトで仮置き(T)化して起草へ進む。
- R2 終了時点で判明分を status: draft + 末尾に未充足観点リストの形で先行保存してよい（以降のラウンドで上書き。セッション切れからヒアリング成果を守る）。

### 3. 起草 — ゲートA（自己チェック）
**スペックは §0 で決めた出力言語で書く（コード・コマンド・slug・技術用語は常に英語）。§1〜12 を埋め、status: draft で即ファイル保存してから提示する**（§13 は実装フェーズで追記）。保存が先・提示が後 — 承認前にセッションが切れてもヒアリング成果を失わないため。
各セクションの根拠は3種のみ: ①本人回答 ②自己調査の結果 ③T仮置き（根拠1行付き）。空欄・「未定」のまま提示するのは規律違反。書くことが本当に無い節は「該当なし」+理由1行（プレースホルダで偽充足しない）。

ゲートAの判定 = **自走チェック**: 「このファイルだけを渡された新規セッションの AI が、追加の質問なしで実装を完成できるか？」観点別の確定の目安:

| § | 観点 | 確定の目安 |
|---|------|-----------|
| 1 | 目的・成功の定義 | 「作ってよかった」と感じる瞬間が一文で言える |
| 2 | 利用者と文脈 | 自分専用/配布の別、利用頻度・場面が分かる |
| 3 | スコープ境界 | v1 機能と「やらないこと」が両方リスト化されている |
| 4 | 入出力と IF | 入力例・出力例・触り口（CLI/GUI/API/自動）が具体化 |
| 5 | 主要フロー | 1回の利用が開始→終了で言語化、中断・多重実行の扱いが決定 |
| 6 | データと永続化 | 残すもの/揮発でよいもの/消えたら困るものが区別済み |
| 7 | 異常系 | 想定外入力時の明示方針（止める・警告して続行・スキップ・リトライ・フォールバック等のどれか）と「絶対に起きてはいけない事故」が明文化 |
| 8 | 非機能 | 性能・秘匿データ・コストが数値/閾値（形容詞のままは未確定扱い） |
| 9 | 環境・制約 | 実行環境・禁止事項・期限（即死条件）が確定 |
| 10 | 既存資産 | 連携/置換/新規独立の別、触ってよい範囲が確定 |
| 11 | 受け入れ基準 | 全項目が実行可能 — コピペで走るコマンド、または番号付き目視手順+期待結果。形容詞だけの AC は不合格 |
| 12 | 裁量範囲 | 仮置きでよい領域と「勝手に決めるな」領域の線引きが確定 |

**ゲートAは自己申告で通過しない。** 次のいずれかを実施する:
- (a) fresh context の汎用 subagent にスペックファイル**だけ**を渡し「これだけで実装するなら何を質問するか」を列挙させる。質問が1つでも返れば未充足 — その項目を潰してから再判定する。
- (b) subagent が使えない環境では、12観点それぞれに該当セクションから根拠行を1つ引用した判定表をチャットに出力する（引用できない観点 = No）。

No になる箇所が残っていれば、それが追加質問（または仮置き）の対象。例外は軽量パスの §12「未確認（ゲートB前に質問予定）」仮記入のみ — 唯一許容される No として通過させ、直後の R3 必須質問で充足してからゲートBに進む。

### 4. 提示と承認 — ゲートB
1. **スペック全文を通常テキストで提示する**（要約でなく全文。AskUserQuestion の本文に長文を詰めない）。
2. AskUserQuestion 1問で確認する: 「このスペックで進めますか？」
   - 選択肢: **「承認して実装開始 (推奨)」「修正したい」「承認のみ（実装は後で）」**
3. 分岐（いずれの承認系分岐でも approved: {YYYY-MM-DD} を記入する）:
   - **修正したい** → ヒアリング全体をやり直さず該当セクションのみ更新し、差分を提示して再ゲートB。
   - **承認のみ** → status: approved でファイル保存し、「実装再開は `/spec specs/<slug>.md`」と案内して終了。
   - **承認して実装開始** → status: implementing で保存し（approved は日付記入のみで通過）、**同一セッションでそのまま実装に入る**。承認済みなので以後フェーズ間の確認はしない。

### 5. 自律実装 — スペックが唯一の指示書
**実装中は迷わず進める。確認するのは次の2つだけ:**
1. **不可逆な決定** — データ削除・外部公開・送金/課金・force push 級、およびユーザーデータ・秘匿情報の外部送信（スペック§8 で許可済みの公開 API への読み取りリクエストは含まない）。
2. **本人にしか決められない新分岐** — スペック§12「勝手に決めるな領域」に該当する事項が実装中に新たに発覚したとき。
例外で質問した場合は、回答を即スペック§12に D-n として記録する（質問が出た事実自体が仕様フェーズの抜けの記録になる）。

それ以外の曖昧さは全て: 最合理仮定を1行で決め、スペック§12に `T-n` として追記し、チャットにも1行表明して前進する。
**仮置きで進めるべき場面で作業を止めて聞くのは規律違反。**

実装の運転規則:
- 破壊的・大規模変更の前に WIP commit。削除はハード削除せず `TRASH/` + `TRASH-FILES.md`。
- multi-GB のモデル・アセットのダウンロードは事前承認（不可逆な決定として扱う）。
- anti-Potemkin: 作った機能は end-to-end で配線確認する（import / 呼び出し / 登録を grep）。built-but-never-called は dead であって done ではない。
- オーケストレーション（subagent fan-out・Workflow 等）は必須ではない。規模に応じた通常判断で選ぶ。
- 進捗・再開点はスペック§13に追記。ユーザーへの報告は問題発生時と完了時のみ。

### 6. 完了 — status 更新と実行証拠
1. **§11 の全 AC を実際に実行する。** コマンド AC は実行し、コマンド・実際の出力（要約でなく末尾数行をそのまま）・exit code を§13に貼る。目視 AC は手順を実施した証拠物（スクリーンショットのパス・観察した具体値）を記録する — 「確認した」だけの記録は未実施扱い。
2. **失敗 AC が残るうちは done にしない。** 直す。直すのに本人決定が要る場合のみ質問する。どうしても通らない AC が残ったら完了と言わない — 何が欠けているかを明示して報告する。
3. 新規ファイル追加・複数ファイル変更・新たな呼び出し配線のいずれかを含む実装は、wiring-auditor subagent（無い環境では汎用 subagent に fresh context で「import/呼び出し/登録を grep して end-to-end 配線を検証せよ」と依頼して代替）の監査を通してから done を宣言する。免除するなら理由1行を§13に書く。
4. status: done + completed 日付に更新する。
5. 完了報告 = ①AC 結果表（各 AC の PASS/FAIL と exit code）②T仮置き一覧の再掲（拒否権をユーザーに渡す）③変更ファイル一覧。
   **「動いたはず」での完了宣言は禁止 — §13に残った実行証拠だけが真実。**

## スペックテンプレート（フォーマット厳守）

```markdown
# {機能名}

- status: draft        <!-- draft | approved | implementing | done -->
- project: {対象プロジェクト root | 新規}
- created: {YYYY-MM-DD} / approved: 未 / completed: 未
- revision: 1

## 1. 目的と成功の定義
{解決する困りごと。完成1週間後に「作ってよかった」と感じる瞬間。}

## 2. 利用者と文脈
{誰が（自分専用/身内配布/公開）・どの頻度で・どんな状況で使うか。}

## 3. スコープ
**やる (v1):**
- {…}
**やらない（明示的スコープ外・将来見送り）:**
- {…}

## 4. 入出力とインターフェース
{入力の形式と実例 / 出力の形式と実例 / 触り口（CLI・GUI・API・自動起動）。}

## 5. 主要フローと状態
{開始→終了の典型1回分。中断・やり直し・多重実行時の挙動。}

## 6. データと永続化
{残すもの / 揮発でよいもの / 消えたら困るものとバックアップ / 保存先。}

## 7. 異常系の方針
{想定外入力: 止める|警告して続行|スキップ|リトライ|フォールバック。環境起因失敗のリトライ方針。絶対に起きてはいけない事故。}

{性能: N秒以内 / 秘匿データの有無と外部送信可否 / コスト上限: 月N円。}
```

## status 遷移
`draft → approved → implementing → done`
- 遷移は必ずファイルの status 行（+日付行）を編集して行う。口頭宣言だけの遷移は無効。「承認して実装開始」では approved は日付記入のみで status は implementing へ直行してよい。
- 起草〜承認の途中で中断（セッション切れ・ユーザー離脱）したら status: draft のまま、未充足の観点をファイル末尾に列挙して保存する（再開時はそこから差分ヒアリング）。
- approved 後・implementing 中に要件変更が出たら（口頭でも改訂扱い）: 既存 D と矛盾しない小さな追加は §12 に D-n を追記し該当セクションを更新して続行。スコープ拡張・既存 D の変更を伴うものは実装を一時停止し、revision +1 で status: draft に戻して再ゲートB（判定: §3「やる(v1)」の項目が増える・§4/§6 の形式や保存対象が増えるならスコープ拡張）。拡張分は v1 に足さず「やらない(将来)」か別スペックへの切り出しを推奨案として提示する — ただし本人が拡張の実施自体を明示要求した場合は本人決定として D-n に記録し取り込んだ案を正とし、見送りは代替案として添えるに留める。
- 実装を中断するときは implementing のまま §13 に再開点を書き残す。

## 改訂（再実行）
`/spec <既存スペックのパス>` または同じ機能名での再実行:
- **status: draft（承認前中断）** → 改訂でなく中断再開。末尾の未充足観点リストから差分ヒアリングを続行する（リストが無ければゲートBから再開。revision 据え置き）。
- **status: approved（未実装）** → 改訂でなく実装再開。AskUserQuestion 1問（選択肢: 「変更なしで実装開始 (推奨)」「変更がある」）で確認し、変更なしなら status: implementing に更新（approved 日付は既存のまま）して手順5へ直行する。変更ありなら下の改訂モードに入る。
- **status: implementing（中断中）** → 改訂でなく実装再開。AskUserQuestion 1問（選択肢: 「変更なしで実装再開 (推奨)」「変更がある」）で確認し、変更なしなら status はそのまま §13 の再開点から手順5を続行する（revision 据え置き）。変更ありなら下の改訂モードに入る。
- **それ以外（done 等）** → 改訂モード: 既存スペックを読み、**D-n（本人決定）は蒸し返さない**。変更・追加点のみ質問する。
  T仮置きの確定（T→D 昇格）はここで行う — 新規質問は起こさず、今回の変更に絡む T のみ個別確認し、残りの T は再ゲートB承認をもって一括昇格とする。
  既存 D と矛盾しない小さな追加は再ゲートBを省略してよい — approved/implementing なら status 遷移節に従い D-n 追記+該当セクション更新で続行。done なら completed を 未 に戻して status: implementing とし、旧 completed 日付と変更内容1行を §13 に保全、追加分の AC を実行してから再度 done にする。
  それ以外（スコープ拡張・既存 D の変更）は revision を +1 し変更内容を1行追記、status を draft に戻して再承認（ゲートB）必須。done からの改訂では completed を 未 に戻す（旧 approved/completed 日付は revision 追記行に保全）。再ゲートBで「承認して実装開始」が選ばれたらそのまま手順5へ。

## 禁止事項
- **調べれば分かることを聞かない。** 「このプロジェクトの言語は？」級は質問枠の浪費であり信頼を失う。
- **決定済みの蒸し返し禁止。** 既知になったことは二度と聞かない。再確認は「私の理解では〜」のプレイバック形式のみ。
- **誘導質問禁止**（「〜という実装でいいですよね？」）。中立な選択肢+推奨+理由で出し、安く拒否できるようにする。
- **白紙のオープン質問禁止**（「要件を教えてください」「他に何かありますか？」）。必ず具体例・選択肢・デフォルト案を添える。
- **1問1論点。** 複合質問は答え漏れの論点を暗黙合意にしてしまう。
- **専門用語で尋問しない。** 「冪等性は必要？」ではなく「同じ操作が2回走っても安全であるべき？」と振る舞いで聞く。
- **HOW から入らない。** 手段の希望は聞いてよいが、必ず目的に遡ってセットで検証する。
- **数値なしの非機能合意禁止。** 秒・件数・円に落ちるまで確定扱いしない。
- **AC 実行証拠なしの done 禁止**（anti-Potemkin）。

調べてから聞け。聞いたら書け。承認されたら走り切れ。
